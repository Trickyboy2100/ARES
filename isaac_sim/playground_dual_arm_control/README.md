## Current Baseline

The current stable baseline is the cuRobo-backed kinematic tray handoff demo:

```bash
/home/andyee/isaacsim/python.sh scripts/gui_tray_handoff_demo.py
```

This script does not start physics playback. cuRobo plans the arm motion segments, then the GUI plays back the verified kinematic trajectory and logs actual scene readback for the gripper pad midpoints and tray pose.

The current tray handoff trajectory is `tray_handoff_curobo_demo/v3`:

- `/World/DemoTray` starts above `/World/LoadingEquip`.
- The tray pose is no longer fixed to world orientation during carry. It is computed every frame from the active gripper pad pose through a calibrated pad-to-tray transform.
- Carry phases use axis-up IK projection so the gripper pad local up axis stays aligned with world Z while the tray follows gripper yaw.
- At the chest handoff, the left and right gripper pad targets are separated by `0.155 m` along world X to account for the tray length. The handoff endpoints were swapped on 2026-06-08: left handoff pad is now `[0.0775, 0.65, 1.2]`, right handoff pad is `[-0.0775, 0.65, 1.2]`, so `right - left = [-0.155, 0, 0]`.
- After the right gripper takes the tray, it moves to a pre-extension pose at tray center `[-0.75, 0.68, 0.952]` with the pad local forward axis constrained to world `-Y`, holds that forward-facing horizontal pose, then extends `0.10 m` forward to the final tray center near `[-0.75, 0.58, 0.952]`.
- cuRobo receives cuboid obstacles for `/World/Table`, `/World/LoadingEquip`, and `/World/Dryer`. The current cuRobo run still prints conservative `Start or End state in collision` warnings for some targets, so the visible carry path is additionally projected and validated before GUI playback.
- The EG2-4C2 grippers now open/close visually during the handoff using the real linkage geometry from `robot-descriptions-common/gripper/inspire_description/xacro/EG2-4C2.xacro` and `jinyu_ros_pkg/nodes/simulation/eg2_4c2.urdf`. The GUI writes `transform:tray_demo_gripper_fk` ops on the flattened visual links: `left_outer_link`, `left_inner_link`, `right_inner_link`, and `right_outer_link` rotate about their URDF pivots, while `left_pad` and `right_pad` are computed through the outer-link pad joints and remain parallel. The imported PhysX gripper articulation is still invalid in this flattened scene, so this is visual kinematic playback rather than dynamic finger joints.
- The clean task scene now contains lightweight simulated Gemini2 wrist cameras on both arms. Their placement is recovered only from the manual Gemini2 transform in `/home/andyee/Documents/isc/202606060821.usd`, relative to each arm's `Link_6`. The old remote Isaac Gemini2 asset reference remains inactive; the usable local camera prims are `.../Gemini2Sim/RgbCamera` and `.../Gemini2Sim/DepthCamera`.

## Next Engineering Direction

1. Keep `gui_tray_handoff_demo.py` as the visible cuRobo-backed baseline.
2. Use `scripts/dual_arm_planning_api.py` as the reusable API layer for external task modules to provide start/end conditions and receive JAKA-ready joint trajectory points.
3. Use `scripts/kinematics_probe.py` as the standard coordinate sanity check.
4. Treat URDF FK/IK as the kinematic source of truth, and treat USD as the world placement/tool-mount source.
5. Keep PhysX off until a separately regenerated nested articulation is needed for dynamics.
6. When dynamics becomes necessary, regenerate the arm USD from URDF instead of driving the flattened imported link layout.

## Planning API

The reusable planning wrapper is:

```text
scripts/dual_arm_planning_api.py
```

It supports:

- per-arm `joint`, `pad_position`, and `pad_pose` goals;
- obstacle inputs from explicit world cuboids or USD prim bounding boxes;
- hard-coded body-midline separation: left arm gripper pads stay at `world X >= 0`, right arm gripper pads stay at `world X <= 0`;
- cuRobo-backed or fallback joint interpolation planning;
- timestamped joint trajectory output with radians/degrees for later JAKA mini 2 SDK adaptation.

Example:

```bash
python3 scripts/dual_arm_planning_api.py \
  --request-json examples/dual_arm_plan_request.json \
  --default-obstacles \
  --out runtime/dual_arm_api_plan_example.json
```

Benchmark:

```bash
python3 scripts/benchmark_dual_arm_planning_api.py --runs 1
python3 scripts/benchmark_dual_arm_planning_api.py --runs 1 --max-attempts 8
python3 scripts/benchmark_dual_arm_planning_api.py --runs 1 --max-attempts 4
python3 scripts/benchmark_dual_arm_planning_api.py --runs 1 --obstacle-preset single-cube
python3 scripts/benchmark_dual_arm_planning_api.py --runs 1 --obstacle-preset none
```

Latest measured planning time:

```text

fallback dual sequential ~= 123 ms
cuRobo single left ~= 5.2 s
cuRobo single right ~= 3.2 s
cuRobo dual sequential ~= 6.4 s
cuRobo max_attempts=8 single left ~= 3.84 s
cuRobo max_attempts=8 single right ~= 1.58 s
cuRobo max_attempts=8 dual sequential ~= 3.30 s
cuRobo with hard midline wall max_attempts=8 dual sequential ~= 3.34 s
cuRobo single-cube obstacle single left ~= 5.37 s
cuRobo single-cube obstacle single right ~= 2.91 s
cuRobo single-cube obstacle dual sequential ~= 5.54 s
cuRobo no-obstacle dual sequential ~= 5.24 s
```

Replacing the current `/World/Table`, `/World/LoadingEquip`, and `/World/Dryer` cuboids with one explicit cube did not make planning much faster. Current timing is dominated by cuRobo planner construction/warmup and pose solve/correction work, not by this small obstacle count.

`ArmPlanRequest.curobo_max_attempts` is exposed for task modules and now defaults to `8`. In the current benchmark, lowering from `20` to `8` roughly halves dual sequential planning time while still returning cuRobo-backed plans; lowering further to `4` or `2` did not materially improve timing for this target pair.

The planning API now always adds side-specific body-midline wall obstacles and validates every output gripper pad trajectory sample. Left-arm pad targets with `world X < 0` and right-arm pad targets with `world X > 0` are rejected before planning.
